\section{Problem Statement}

Corporate commuting represents a significant challenge, particularly in organizations where employees travel daily to a shared workplace. The widespread use of private vehicles leads to increased traffic congestion, higher transportation costs, and a considerable environmental impact.

Although carpooling offers a promising solution, existing approaches remain limited in their effectiveness. The main issue lies in the lack of flexibility and intelligence in matching users, especially when their locations and schedules vary.

Thus, the problem addressed in this project can be formulated as follows:

\textit{How to design an intelligent, flexible, and scalable carpooling system capable of efficiently matching users within a real-world enterprise environment?}

\section{Project Objectives}

The main objective of this project is to design and implement an intelligent carpooling platform tailored for enterprise environments.

To achieve this goal, several specific objectives were defined:

\begin{itemize}
    \item Develop a flexible ride-sharing system adapted to employee commuting patterns
    \item Design an intelligent matching mechanism based on route overlap
    \item Integrate machine learning techniques for commute prediction
    \item Ensure real-time interaction between users
    \item Provide a scalable and maintainable system architecture
\end{itemize}

These objectives guided the design and implementation decisions throughout the project.

\section{Analysis of Existing Solutions}

Several carpooling platforms have been developed to address transportation challenges, including widely known solutions such as BlaBlaCar and Uber Pool. These platforms aim to connect drivers and passengers through shared rides and are primarily designed for public use.

However, in the context of this project, there are no direct competitors. The proposed system is intended for internal use within a single enterprise, where all users belong to the same organization. This creates a fundamentally different operational environment compared to public platforms.

Therefore, existing solutions such as BlaBlaCar and Uber Pool are considered indirect competitors. While they provide valuable insights into ride-sharing mechanisms, they are not adapted to enterprise-specific constraints.

These platforms typically rely on predefined routes and fixed pickup locations, limiting flexibility. Moreover, they operate in open environments where trust between users is not guaranteed and must be managed through rating systems and external verification mechanisms.

In contrast, an enterprise environment inherently ensures a higher level of trust, as all users are verified employees within the same organization.

\section{Limitations of Existing Solutions}

The analysis of existing systems highlights several limitations when applied to enterprise contexts:

\begin{itemize}
    \item Rigid matching mechanisms based on fixed pickup locations
    \item Limited flexibility in accommodating diverse user routes
    \item Lack of adaptation to recurring commuting patterns
    \item Dependence on external trust mechanisms
    \item Absence of enterprise-oriented design and coordination features
\end{itemize}

These limitations significantly reduce the effectiveness of traditional carpooling solutions in structured organizational environments.

\section{Proposed Solution}

To address these challenges, this project proposes an intelligent carpooling platform specifically designed for internal enterprise use.

Unlike public carpooling platforms, the proposed system operates within a closed organizational environment. This provides several key advantages, including enhanced privacy, simplified user verification, and a higher level of trust between users.

The system introduces a dynamic matching mechanism based on route overlap analysis. Instead of relying on fixed pickup points, passengers can be matched along a driver's route, significantly increasing matching opportunities.

In addition, a machine learning component is integrated to analyze historical commuting data and identify recurring patterns. This enables predictive capabilities and improves user engagement through proactive suggestions.

By combining deterministic geospatial algorithms with data-driven prediction, the system achieves a balance between precision, flexibility, and adaptability.

This approach fundamentally departs from conventional carpooling systems, which rely on fixed-point matching and passive user interaction. By integrating route-based trajectory analysis with predictive intelligence and real-time event coordination, the platform transforms carpooling from a static search problem into a dynamic, context-aware mobility system. This architectural integration of deterministic matching, machine learning, and event-driven synchronization constitutes the core originality of the proposed solution.

\subsection{Advantages of the Proposed Solution}

\section{Technologies Used}

The development of the proposed system relied on a carefully selected technology stack. Each technology was chosen based on specific criteria, including performance, scalability, ease of integration, and suitability for the system requirements.

\subsection{Backend Technology}

The backend was implemented using \textbf{Flask}, a lightweight Python web framework.

\begin{figure}[H]
    \centering
    \includegraphics[width=3cm]{images/logos/flask.pdf}
    \caption{Flask Framework Logo}
\end{figure}

Flask was chosen due to its simplicity, flexibility, and ability to support modular development. Unlike more complex frameworks, Flask allows precise control over system architecture and facilitates the integration of custom logic, which is essential for implementing the intelligent matching system.

Additionally, Python provides strong support for data processing and machine learning, making it particularly suitable for integrating predictive components into the system.

\subsection{Frontend Technology}

The frontend was developed using \textbf{Flutter}, a cross-platform framework for building mobile applications.

\begin{figure}[H]
    \centering
    \includegraphics[width=3cm]{images/logos/flutter.pdf}
    \caption{Flutter Framework Logo}
\end{figure}

Flutter was selected because it enables the development of a single codebase that can run on multiple platforms, ensuring consistency and reducing development time. Its reactive UI model allows the creation of responsive and dynamic user interfaces, which is essential for real-time interaction within the application.

\subsection{Database Management System}

The system uses \textbf{PostgreSQL} as its database management system.

\begin{figure}[H]
    \centering
    \includegraphics[width=4cm]{images/logos/postgresql.pdf}
    \caption{PostgreSQL Logo}
\end{figure}

PostgreSQL was chosen for its reliability, scalability, and advanced querying capabilities. It also provides strong support for handling structured data and ensures data integrity, which is critical for managing user information and ride data.

\subsection{Geospatial Services}

Geospatial processing is handled using \textbf{OpenStreetMap (OSM)} and \textbf{OSRM (Open Source Routing Machine)}.

\begin{figure}[H]
    \centering
    \includegraphics[width=2cm]{images/logos/Openstreetmap_logo.pdf}
    \caption{OpenStreetMap Logo}
\end{figure}

These tools were selected because they provide accurate route computation and allow efficient processing of geographic data. OSRM, in particular, enables fast route calculations, which are essential for implementing the route-based matching system.

\subsection{Real-Time Communication}

Real-time interaction between users is enabled through \textbf{WebSocket} technology.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.15\textwidth]{images/logos/websocket.pdf}
    \caption{WebSocket Communication Technology Logo}
    \label{fig:websocket}
\end{figure}

WebSockets allow continuous bidirectional communication between the client and the server, making them particularly suitable for real-time applications. Unlike traditional HTTP requests, which follow a request-response model, WebSockets maintain an open connection that enables instant data exchange.

In the context of this system, WebSocket communication is used to provide real-time notifications, ride status updates, and synchronization between users. This ensures that passengers and drivers receive immediate feedback when events occur, such as ride confirmations, cancellations, or changes in availability.

The use of WebSockets significantly enhances user experience by reducing latency and enabling a dynamic and responsive interaction model within the platform.
\subsection{Technology Selection Justification}

The chosen technology stack ensures a balance between performance, scalability, and flexibility. It supports the integration of intelligent components, real-time features, and geospatial processing, which are core aspects of the proposed system.

This combination of technologies provides a solid foundation for building a robust, maintainable, and extensible enterprise carpooling platform.
\section{Development Methodology}

The project followed an Agile development methodology based on the Scrum framework. Scrum structures work into time-boxed iterations called sprints, typically lasting two to four weeks, delivering potentially shippable product increments.

This methodology was selected due to its suitability for the project context:
\begin{itemize}
    \item \textbf{Iterative development:} Progressive system construction with regular validation
    \item \textbf{Continuous feedback:} Stakeholder input throughout the development lifecycle
    \item \textbf{Incremental delivery:} Functional features at regular intervals
    \item \textbf{Flexibility:} Accommodation of evolving requirements
    \item \textbf{Risk mitigation:} Early issue detection through frequent reviews
\end{itemize}

The main concepts of the Scrum methodology are illustrated in Figure~\ref{fig:scrum}.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.75\textwidth]{images/methodology/scrum.jpg}
    \caption{Scrum Development Process}
    \label{fig:scrum}
\end{figure}

The methodology was applied through sprint planning, daily progress tracking, sprint reviews, and retrospectives. Validation activities were performed continuously at the end of each sprint, ensuring consistent quality and alignment with project objectives.

\section{Scrum Roles}

The Scrum framework defines three core roles, adapted to this professional internship context as follows:

\begin{itemize}
    \item \textbf{Product Owner:} The industrial supervisor from the host company, responsible for representing business stakeholders, defining project requirements, validating system increments, and ensuring the delivered solution creates measurable value for the enterprise.
    \item \textbf{Scrum Master:} The process facilitator responsible for organizing sprint planning and review sessions, monitoring progress, removing technical and organizational impediments, and ensuring strict adherence to Agile practices throughout the development lifecycle.
    \item \textbf{Development Team:} A collaborative development team composed of the intern working in coordination with technical supervisors and company mentors. This structure ensures continuous validation, code review, and knowledge transfer within a real engineering environment, while maintaining cross-functional responsibility for design, implementation, and quality assurance of each system increment.
\end{itemize}

\section{Definition of Done}

A Definition of Done (DoD) was established to ensure consistent quality standards. A feature is considered complete when:

\begin{itemize}
    \item \textbf{Implemented:} Functionality meets acceptance criteria
    \item \textbf{Tested:} Unit and integration tests pass with adequate coverage
    \item \textbf{Validated:} Demonstrated and reviewed with the Product Owner
    \item \textbf{Integrated:} Backend, frontend, and database layers work together
    \item \textbf{Documented:} Technical documentation and code comments provided
\end{itemize}

This DoD ensures each increment is potentially shippable and quality-assured, maintaining engineering standards while enabling continuous delivery.
\section{Global Architecture Overview}

The system follows a layered architecture composed of three main components:

\begin{itemize}
    \item \textbf{Frontend Layer:} Handles user interaction through a mobile application
    \item \textbf{Backend Layer:} Manages business logic, APIs, and processing
    \item \textbf{Data Layer:} Stores and manages system data using a relational database
\end{itemize}

This architecture ensures separation of concerns, scalability, and maintainability.

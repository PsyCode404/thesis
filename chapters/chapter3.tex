\chapter{User Management and Authentication} 

\section{Introduction}

This chapter describes the implementation of user management and authentication functionalities, which form the foundation of the platform. These components control access to the system, enforce enterprise-specific restrictions, and structure how users interact with the application.

The implementation corresponds to \textbf{Sprint 1} within the Agile development process. This sprint focuses on establishing a reliable authentication mechanism based on enterprise credentials, managing user profiles, and providing administrative control features. Each functionality was developed incrementally and validated at the end of the sprint to confirm correct behavior and system stability.

\section{Sprint 1: User Management and Authentication}

\subsection{Sprint Goal}

The goal of this sprint is to implement a secure access mechanism based on enterprise authentication while providing essential user management capabilities. This includes profile handling, role-based access control, and administrative supervision features.

This sprint is critical because it establishes the security backbone of the platform and guarantees that only authorized users can access its functionalities.

    \caption{Authentication Sequence Diagram}

The sequence diagram describes the interaction between system components during the authentication process. It highlights the integration between the mobile application, the backend, and the external identity provider.

The authentication workflow proceeds as follows:

\begin{enumerate}
    \item The user initiates authentication from the mobile application.
    \item The application redirects the request to the Microsoft Office 365 authentication service.
    \item After successful authentication, an access token is returned to the client.
    \item The token is sent to the backend server for verification.
    \item The backend validates the token and applies domain-based access control.
    \item If the email domain is valid, the system retrieves or creates the corresponding user record.
    \item A secure session is established, and access to the platform is granted.
\end{enumerate}

This approach separates identity verification from application logic. Authentication is handled by a trusted external provider, while access restrictions are enforced internally, ensuring both security and organizational control.

\section{Authentication Mechanism}

Authentication is integrated with Microsoft 365 enterprise accounts to provide Single Sign-On (SSO) capabilities. After successful authentication, the backend generates JSON Web Tokens (JWT)~\cite{jwt} used to secure subsequent API communication between the mobile application and backend services.

Access is strictly limited to users with a valid professional email address within the \textbf{@gexpertise.fr} domain. This restriction guarantees that only authorized employees can access the platform.

From an architectural perspective, authentication is divided into two stages:

\begin{itemize}
    \item \textbf{External Authentication:} Managed by Microsoft 365, responsible for verifying user identity through enterprise credentials.
    \item \textbf{Internal Token Management:} Performed by the backend, responsible for JWT generation, validation, and enforcing domain restrictions.
\end{itemize}

This two-step approach combines trusted external identity verification with application-level control, strengthening overall security.

\subsection{Enterprise Access Control Model}

The access control model is designed specifically for enterprise environments where user membership is restricted to a defined organization.

Unlike public platforms that allow open registration, the system enforces a closed user model in which access depends on possession of a valid corporate identity.

This design has several implications. First, the user base is limited to employees of the organization, removing the need for external trust mechanisms such as rating systems or third-party verification. Second, user creation is not initiated manually but triggered automatically after successful authentication. The system either retrieves an existing record or creates a new one during the first login, ensuring that all users originate from a validated identity source. Third, the platform applies role-based access control, distinguishing between collaborators and administrators. Administrative roles provide additional capabilities such as account management, activity monitoring, and policy enforcement.

This model maintains a controlled user population, simplifies access governance, and aligns the platform with enterprise operational requirements.

\section{Profile Management}

Authenticated users can access and manage their personal profiles, which support personalization and interaction within the platform.

Users are able to:

\begin{itemize}
    \item View personal and professional information
    \item Update profile data such as contact details and vehicle information
\end{itemize}

All updates are validated at the backend before being stored in the database. This validation step maintains data consistency, prevents invalid entries, and preserves the reliability of user information.

\section{Administrative Features}

Administrators play a key role in maintaining platform stability and enforcing operational policies. Their responsibilities go beyond standard user interaction and include supervision and control mechanisms.

The platform provides the following administrative capabilities:

\begin{itemize}
    \item Monitoring user activity and overall system usage
    \item Identifying abnormal or suspicious behavior
    \item Freezing or restricting user accounts when required
\end{itemize}

These capabilities maintain a controlled environment aligned with enterprise requirements. In particular, the ability to restrict accounts is essential for handling misuse and enforcing organizational policies.

\section{Sprint Validation}

At the end of Sprint 1, a structured validation process was conducted to verify that all implemented functionalities satisfy both functional and security requirements. Following the Agile approach, validation is performed incrementally at the end of each sprint to confirm system reliability before moving to subsequent phases.

\subsection{Validation Objectives}

The validation process focuses on the following objectives:

\begin{itemize}
    \item Verify correct behavior of authentication and access control mechanisms
    \item Confirm enforcement of enterprise domain restrictions
    \item Validate user profile management operations
    \item Ensure proper functioning of administrative control features
\end{itemize}

\subsection{Test Scenarios}

The system was evaluated through a set of representative functional scenarios:

\begin{itemize}
    \item \textbf{Scenario 1: Successful Authentication} \\
    A user attempts to log in using a valid Office 365 account associated with an authorized domain. The system authenticates the user and grants access to the platform.

    \item \textbf{Scenario 2: Unauthorized Domain Access} \\
    A user attempts to authenticate with an email address outside the \textbf{@gexpertise.fr} domain. The system rejects the request and displays an appropriate error message.

    \item \textbf{Scenario 3: Profile Update Validation} \\
    An authenticated user updates profile information. The system validates the input and persists the updated data in the database.

    \item \textbf{Scenario 4: Administrative Control} \\
    An administrator performs actions such as monitoring activity or restricting an account. The system applies these changes consistently and reflects them immediately.
\end{itemize}

\subsection{Expected Results}

Each scenario corresponds to the following expected outcomes:

\begin{itemize}
    \item Authorized users are authenticated and granted access to the application
    \item Unauthorized users are denied access based on domain validation rules
    \item Profile updates are validated and correctly stored
    \item Administrative actions are applied and reflected in platform behavior
\end{itemize}

\subsection{Validation Outcome}

All defined scenarios were executed successfully, confirming that the implemented functionalities behave as expected under both standard and exceptional conditions.

The validation results demonstrate that authentication, user management, and administrative control features operate correctly and align with system requirements.

In addition, the tested scenarios are consistent with the interactions illustrated in the sequence diagram, confirming alignment between system design and implementation.

\section{Conclusion}

This chapter presented the implementation of user management and authentication functionalities, which constitute the foundational security layer of the platform. The developed mechanisms establish controlled access to the system by integrating enterprise-level authentication with application-level validation.

The use of Single Sign-On (SSO) through Microsoft Office 365 enables secure and seamless user identification while eliminating the need for manual account creation. This approach ensures that all users are authenticated through a trusted external provider, while internal validation enforces domain-based restrictions aligned with organizational boundaries.

In addition, the implementation of a role-based access control model allows the system to differentiate between standard users and administrators, ensuring that each actor operates within clearly defined permissions. Administrative features provide the necessary tools to monitor activity, manage user accounts, and maintain overall platform integrity.

The profile management module further enhances user interaction by allowing controlled updates of personal data, while backend validation ensures consistency and reliability of stored information. The validation process conducted at the end of the sprint confirms that all functionalities behave as expected under both normal and exceptional scenarios.

Overall, this chapter establishes a robust and secure user management framework that supports reliable system operation. It lays the groundwork for higher-level functionalities by ensuring that all interactions within the platform are authenticated, controlled, and consistent.

The next chapter builds upon this foundation by introducing the implementation of ride management and real-time communication features, which extend the system toward dynamic and interactive use cases.

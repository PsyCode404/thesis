\chapter{System Architecture and Design}

\section{Introduction}

This chapter presents the architectural design of the proposed intelligent carpooling platform. It describes how the system is organized, how its components interact, and the main design principles that guided its development.

The objective is to provide a clear understanding of the platform's structure before implementation. The chapter introduces a global architectural view, details the main components, and presents modeling through diagrams. It also explains how data flows across layers to support different functionalities.

\section{Global System Architecture}

The architectural foundation of the system was established during \textbf{Sprint 0}, which focused on environment setup, project initialization, and defining the global system structure. This phase enabled the identification of core components and their interactions, forming the basis for subsequent development.

The system follows a layered architecture composed of three main components: the frontend, the backend, and the data layer. This separation improves modularity, scalability, and maintainability by clearly defining responsibilities.

\begin{itemize}
    \item \textbf{Frontend Layer:} A mobile application developed using Flutter, responsible for handling user interaction and rendering the interface.
    
    \item \textbf{Backend Layer:} A server implemented using Flask, acting as the central processing unit of the system. It manages business logic, intelligent matching, and data processing.
    
    \item \textbf{Data Layer:} A PostgreSQL database used to store, manage, and retrieve system data efficiently.
\end{itemize}

The interaction between these components is illustrated in Figure~\ref{fig:global_architecture}.

\begin{figure}[htbp]

    \caption{Global System Architecture}

The mobile application serves as the entry point for user interaction. It communicates with the backend through RESTful APIs using HTTP requests and receives structured JSON responses. In addition, real-time communication is achieved through WebSocket connections, allowing instant updates such as notifications and ride status changes.

The backend represents the core processing layer of the system. It handles multiple responsibilities, including request processing, business logic execution, geospatial computations, and decision-making. Centralizing these operations guarantees consistency and efficient coordination across components.

The backend interacts with the PostgreSQL database using SQL queries to manage persistent data. It also communicates with external geospatial services, particularly OSRM~\cite{osrm}, to obtain routing and distance information required for the matching process.

This architecture provides a clear separation of concerns while maintaining efficient communication between components, enabling scalability and progressive evolution.

\section{Architectural Design Decisions}

Several key design decisions were made so that the platform satisfies both functional and non-functional requirements.

\subsection{Layered Architecture}

A layered architecture was adopted to separate responsibilities between the user interface, business logic, and data management layers. This design improves maintainability and allows each layer to evolve independently without affecting the rest of the architecture.

\subsection{Client-Server Model}

The system follows a client-server model, where the frontend communicates with the backend through RESTful APIs~\cite{rest}. This approach supports scalability and allows multiple clients to interact with the backend simultaneously.

\subsection{Technology Selection Rationale}

The choice of technologies was driven by system requirements and constraints. Flask was selected for its flexibility in implementing backend logic, Flutter for its ability to support cross-platform mobile development, and PostgreSQL for its robustness and efficient data management capabilities.

\subsection{Architectural Trade-offs and Design Justification}

The architectural choices presented in this chapter are the result of deliberate engineering decisions based on the project scope, operational constraints, and long-term maintainability goals. These decisions aim to balance performance, scalability, flexibility, and ease of future evolution.
\subsubsection{Monolithic Architecture over Microservices}

A modular monolithic architecture was selected instead of adopting a microservices-based approach. Although microservices provide advantages such as independent scalability and deployment flexibility, they also introduce additional complexity related to service orchestration, inter-service communication, and distributed data consistency.

Given the scope of this project---an enterprise application with a controlled number of users---these complexities were not justified. The operational overhead required to manage multiple services would outweigh the practical benefits in this context.

The chosen monolithic architecture is internally organized into well-defined modules, including authentication, ride management, matching, real-time communication, and prediction. This modular structure preserves logical separation while maintaining a unified deployment model.

As a result, the system benefits from simplified deployment, easier debugging, and strong transactional consistency without requiring distributed coordination mechanisms.

\subsubsection{Backend-Centric Business Logic Enforcement}

All core business logic is implemented and enforced within the backend, while the frontend is limited to presentation and user interaction.

This design choice is motivated by several factors. First, it guarantees consistency, as all application rules are executed in a centralized environment, yielding identical behavior across different client platforms. Second, it improves security by preventing users from bypassing constraints through client-side manipulation, which is particularly important for operations such as authentication, booking validation, and administrative control. Third, it enhances maintainability, since updates to business rules can be performed directly on the server without requiring coordinated changes across multiple clients.

This backend-centric approach keeps the frontend lightweight and focused on user experience, while reinforcing the reliability and integrity of the system.

\subsubsection{REST and WebSocket Hybrid Communication}

The system adopts a hybrid communication model that combines RESTful APIs with WebSocket connections.

REST is used for standard request-response interactions such as authentication, profile management, ride creation, and data retrieval. In contrast, WebSocket~\cite{websocket} is dedicated to real-time, event-driven communication.

This separation allows each communication method to be used where it is most effective. Polling-based solutions, although simpler, generate unnecessary network traffic and introduce latency when frequent updates are required. WebSocket addresses this limitation by maintaining a persistent bidirectional connection, enabling the backend to push updates instantly to connected clients.

By combining these two mechanisms, the platform achieves efficient resource utilization while maintaining fast and synchronized updates across all users.

\subsubsection{Centralized Geospatial Processing}

All geospatial computations are performed on the backend, including route analysis, corridor generation, and detour estimation, using OSRM-based services.

Centralizing these operations guarantees that matching results remain consistent regardless of variations in client-side computation or device capabilities. It also prevents discrepancies caused by differences in user devices or location precision.

In addition, this design allows full control over the evolution of matching algorithms. Any improvement to routing strategies or scoring mechanisms can be deployed directly on the server without requiring updates to the client application.

Finally, keeping geospatial processing on the backend reduces the computational load on mobile devices and protects the internal logic of the matching system.

\section{System Components Description}

\subsection{Frontend Layer}

The frontend is implemented as a mobile application using Flutter. It provides the user interface and manages all interactions, including ride creation, ride search, booking operations, and notifications.

The frontend communicates with the backend through HTTP requests and maintains real-time synchronization using WebSocket connections.

\subsection{Backend Layer}

The backend acts as the central processing unit of the system. It is responsible for receiving client requests, executing business logic, and coordinating communication between system components.

It integrates multiple functional modules, including:
\begin{itemize}
    \item REST API layer for handling client requests
    \item Core business logic implementation
    \item Geospatial matching engine
    \item Machine learning prediction module
\end{itemize}

\subsection{Data Layer}

The data layer is implemented using PostgreSQL. It manages persistent storage of system data, including users, rides, reservations, and system interactions.

The database maintains data consistency and integrity through relational constraints and structured schema design. It also supports efficient querying and data retrieval.

Additionally, PostgreSQL provides advanced capabilities such as indexing and query optimization, which improve system performance when handling large datasets.

\section{Data Flow Description}

A typical interaction within the system follows a structured sequence of operations:

\begin{enumerate}
    \item A collaborator initiates an action through the mobile application.
    \item The frontend sends an HTTP request to the backend.
    \item The backend processes the request by applying business logic and matching rules.
    \item If required, the backend retrieves or stores data in the PostgreSQL database.
    \item The backend communicates with OSRM to perform geospatial computations.
    \item The result is returned to the frontend in the form of a JSON response.
    \item The frontend updates the user interface based on the received data.
    \item Real-time updates are transmitted through WebSocket connections when necessary.
\end{enumerate}

This workflow enables efficient communication between layers and supports both synchronous and real-time interactions.

\section{Integration of Intelligent Components}

The system integrates two complementary intelligent mechanisms.

The first component is a deterministic matching system based on geospatial analysis. It identifies suitable ride matches by analyzing route overlap and real-time constraints.

The second component is a machine learning module designed to analyze historical (or synthetic) data in order to detect commuting patterns and generate predictive suggestions.

Both components are integrated within the backend while remaining logically separated. This design preserves system stability while allowing each component to evolve independently. A detailed description of these intelligent mechanisms is provided in Chapter~5.

\section{Communication and Security}

Communication between system components relies on a combination of RESTful APIs and WebSocket protocols.

REST APIs are used for standard request-response interactions, such as authentication, data retrieval, and ride management operations. In contrast, WebSocket connections are used to handle real-time communication, enabling instant updates for events such as ride status changes and notifications.

Security is enforced through multiple mechanisms, including user authentication, role-based access control, and controlled data exchange between layers. These measures guarantee that only authorized users can access functionalities and perform sensitive operations.

\section{Scalability and Maintainability Considerations}

The system architecture was designed to support future growth while remaining easy to maintain.

The layered structure allows each component to evolve independently without impacting the entire system. The use of APIs facilitates the integration of additional clients or services, while the modular backend design simplifies the introduction of new features or improvements.

This approach allows the platform to adapt to changing requirements without requiring significant architectural modifications.

\section{System Constraints}

The system operates under several constraints that influenced its design choices:

\begin{itemize}
    \item Dependency on external geospatial services (OSRM), which introduces reliance on third-party availability and response time
    \item Network latency that may affect real-time communication performance
    \item Mobile device limitations, including battery consumption and variable connectivity conditions
    \item Limited availability of real-world historical data for training machine learning models
\end{itemize}

These constraints were considered during design to maintain reliability and acceptable performance under realistic conditions.

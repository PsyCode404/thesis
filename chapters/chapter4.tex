\chapter{Ride Management and Real-Time Interaction}

\section{Introduction}

Following the implementation of authentication and user management, this chapter introduces the operational core of the platform: ride management and real-time interaction.

This layer coordinates all ride-related operations while maintaining consistency, enforcing constraints, and synchronizing system state across all components.

Two complementary concerns are addressed:

\begin{itemize}
    \item \textbf{State Management}: handling ride creation, booking processes, and lifecycle transitions under strict business rules.
    \item \textbf{State Propagation}: ensuring that all participants observe a consistent and up-to-date system state in real time.
\end{itemize}

To address these concerns, the implementation is divided into two development phases:

\begin{itemize}
    \item \textbf{Sprint 2 — Ride Management}: focuses on entity modeling, constraint enforcement, and lifecycle management.
    \item \textbf{Sprint 3 — Real-Time Interaction}: introduces an event-driven communication layer for live synchronization.
\end{itemize}

This separation distinguishes decision logic from data propagation, improving modularity and simplifying system evolution.

\subsection{Sequence Diagram — Ride Booking}

The booking process is implemented as a controlled validation sequence executed entirely on the backend.

Although the request originates from the client, all decision-making logic remains centralized on the server, preventing any bypass of constraints.

The validation process follows a defined sequence:

\begin{itemize}
    \item Retrieval of the ride from the database to confirm its existence.
    \item Verification of seat availability to avoid overbooking.
    \item Enforcement of role constraints, preventing users from booking their own rides.
    \item Validation of temporal conditions to ensure the ride is still active.
\end{itemize}

Each step acts as a validation checkpoint. If any condition fails, the process stops immediately.

Only when all validations succeed does the system create the reservation and update seat availability, ensuring consistency.

\subsection{Real-Time Architecture}

To achieve immediate synchronization between distributed system components, the platform adopts a communication architecture based on WebSocket technology~\cite{websocket, pimentel2012websocket}.

Unlike traditional HTTP communication, which follows a request-response paradigm, WebSocket establishes a persistent bidirectional connection between the client and the server. Once initialized, this connection remains continuously active, enabling the backend to transmit updates asynchronously without requiring repeated polling requests from clients.

This architectural approach is particularly suited to ride-sharing environments, where system state evolves continuously and multiple users may need to observe the same information simultaneously.

The adoption of WebSocket communication is motivated by several technical requirements:

\begin{itemize}
    \item low-latency interaction between distributed clients,
    \item continuous synchronization of ride lifecycle state,
    \item efficient propagation of high-frequency events such as location updates,
    \item reduction of unnecessary network overhead generated by repetitive polling.
\end{itemize}

By maintaining persistent communication channels, the system avoids the delays and inefficiencies associated with repeatedly establishing new HTTP requests for each update.

% -----------------------------------------------------

\subsubsection{Connection Lifecycle}

The real-time communication workflow begins when the mobile application establishes a WebSocket connection with the backend server.

This connection is linked to the authenticated user session, ensuring that events are transmitted only to authorized and relevant participants. The authentication context therefore remains integrated into the communication layer itself.

Once the connection is established, it remains active throughout the ride lifecycle, enabling continuous interaction between the backend and connected clients without additional request overhead.

The persistent nature of the connection is particularly important during active rides, where state transitions and location changes occur frequently and require near real-time propagation.

In the event of temporary network interruption, reconnection mechanisms are used to restore communication and resynchronize the client with the latest backend state. This prevents prolonged inconsistencies between users and ensures continuity of interaction.

% -----------------------------------------------------

\subsubsection{Event-Driven Communication Model}

The platform follows an event-driven communication model in which significant backend actions generate events transmitted through the WebSocket layer.

Whenever an important change occurs within the system, such as:

\begin{itemize}
    \item ride status updates,
    \item driver location updates,
    \item reservation confirmations,
    \item notification triggers,
\end{itemize}

the backend immediately emits an event corresponding to the updated system state.

Clients subscribed to the associated ride or user channel receive these events in real time and update their interface dynamically.

This mechanism transforms the platform from a request-oriented system into a reactive environment where interfaces evolve continuously according to backend activity.

The event-driven approach also removes the need for periodic polling mechanisms, reducing unnecessary traffic while improving responsiveness and synchronization consistency across users.

% -----------------------------------------------------

\subsubsection{Multi-Client Synchronization}

An essential property of the adopted architecture is its ability to synchronize multiple connected clients simultaneously.

For a single ride session, several participants may remain connected concurrently, including the driver and multiple passengers. Whenever a relevant event is emitted, the updated information is propagated to all concerned users in parallel.

This synchronization mechanism ensures that:

\begin{itemize}
    \item ride status remains identical across all devices,
    \item location changes are reflected simultaneously for connected participants,
    \item user interfaces remain continuously aligned with backend state,
    \item interactions performed by one participant become immediately visible to others.
\end{itemize}

As a result, all users share a coherent and continuously updated representation of ride execution, reducing the risk of inconsistent information or delayed interaction feedback.

\subsection{Notification System}

The notification system complements the real-time layer by transforming backend events into structured user-facing information.

While WebSocket enables immediate event delivery, the notification layer provides persistence and contextual representation of these events.

\subsubsection{Role of Notifications}

Notifications serve several purposes:

\begin{itemize}
    \item Inform users about important ride-related events in real time
    \item Provide immediate feedback on system actions
    \item Guide users toward relevant interactions within the application
    \item Maintain awareness even when the user is not actively viewing the ride
\end{itemize}

% ------------------------
\subsubsection{System Flow}

The notification mechanism follows a structured processing pipeline:

\begin{enumerate}
    \item A system event occurs (e.g., reservation accepted, ride status updated)
    \item The backend generates and stores a notification entry in the database
    \item A WebSocket event is emitted to the concerned user(s)
    \item The mobile application receives the event in real time
    \item The notification is displayed and the interface is updated
\end{enumerate}

This dual mechanism ensures both persistence through database storage and immediacy through real-time delivery.

% ------------------------
\subsubsection{Types of Notifications}

The platform supports multiple categories of notifications, each corresponding to a specific type of system event:

\begin{itemize}
    \item Ride lifecycle updates (ride started, driver arrived, ride completed)
    \item Reservation updates (accepted or rejected requests)
    \item Boarding alerts (upcoming or missed boarding)
    \item System alerts (ride cancellation, availability changes)
\end{itemize}

These categories ensure that users receive relevant information depending on their role and current context within the ride.

Each notification includes contextual data such as identifiers, timestamps, and associated ride information, enabling precise interaction and navigation within the application.

% ------------------------
\subsubsection{User Interaction and Navigation}

Notifications are designed as interactive elements rather than passive messages, allowing users to respond directly to system events.

Users can:

\begin{itemize}

    \item Mark notifications as read individually or in bulk
    \item Navigate directly to relevant application screens by interacting with notifications
\end{itemize}

This interaction model reduces the need for manual navigation and allows users to quickly access the most relevant information.

The navigation behavior depends on the notification context:

\begin{itemize}
    \item Ride-related notifications redirect to the \textit{Ride Details Screen}
    \item Reservation-related notifications redirect to the \textit{Reservation Details Screen}
    \item Boarding or pickup notifications redirect to the active ride interface
\end{itemize}

By linking notifications to specific application states, the system ensures that users can move seamlessly from an event to the corresponding action.

\subsection{System Observability and Administrative Interface}

In event-driven systems, where behavior emerges from asynchronous interactions between distributed components, observability becomes essential for maintaining system reliability~\cite{event_driven_architecture}.

Without explicit traceability mechanisms, diagnosing race conditions, identifying inconsistencies, and analyzing event propagation becomes significantly more complex.

In addition to user-facing functionalities, the platform incorporates an administrative observability layer that provides real-time visibility into system activity and operational state.

This layer is not implemented as a separate module, but rather as an extension of the event-driven architecture described earlier. It leverages the same real-time infrastructure to expose system activity, enabling administrators to monitor and analyze system behavior as it occurs.

\subsubsection{System Event Logging}

At the core of this layer lies a structured event logging mechanism.

Each significant action within the system—such as user authentication, ride creation, reservation updates, and lifecycle transitions—is recorded as a \textit{system event}. These events are generated at the backend and stored in a dedicated data model, ensuring traceability and historical consistency.

This approach ensures that system interactions are captured in a standardized format, allowing precise tracking of both user activity and system behavior over time.

\subsubsection{System Transition to Real-Time Coordination}

The introduction of the real-time communication layer represents a fundamental architectural shift for the platform.

Prior to this stage, the system followed a traditional request-response model, where the client initiated an action and the backend returned a result. While suitable for transactional operations such as authentication or profile updates, this model does not support the coordination requirements of dynamic ride execution.

With the integration of a WebSocket-based event infrastructure, the platform transitions to an event-driven coordination model~\cite{event_driven_architecture}. In this model, the backend acts as an event dispatcher, continuously observing state changes and propagating updates to relevant clients. The frontend becomes a reactive component that receives updates asynchronously and reflects them immediately in the interface.

This transition introduces several key improvements. First, consistency: all participants observe the same state transitions simultaneously, reducing the risk of inconsistent views. Second, interaction efficiency: users are no longer required to manually refresh or request updates. Third, coordination: the backend maintains full control over ride execution, acting as the single source of truth across all connected clients.

The real-time system was validated through multi-client scenario testing conducted on the integrated platform.

The following conditions were verified:

\begin{itemize}
    \item simultaneous connections receiving synchronized updates,
    \item real-time propagation of ride status and location updates,
    \item correct system behavior under connection interruptions,
    \item consistency of state after client reconnection,
    \item proper delivery and interaction of notifications.
\end{itemize}

These tests confirm that the system maintains reliable synchronization, consistent state propagation, and responsive user interaction under dynamic conditions.

\section{Conclusion}

This chapter presented the implementation of a complete ride management system enhanced by a real-time communication architecture.

The integration of backend-controlled logic, a structured ride lifecycle, and a WebSocket-based event-driven model ensures both consistency and responsiveness across the platform. Each component, from ride state transitions to user-facing notifications, is tightly synchronized, allowing all participants to maintain a shared and up-to-date view of ongoing rides.

The introduction of the notification system further strengthens this architecture by transforming system events into actionable interactions, effectively linking backend processes with frontend behavior.

Overall, the system provides a robust and reactive environment capable of supporting dynamic ride execution scenarios while preserving data integrity and user awareness.

However, despite its responsiveness, the system remains fundamentally reactive. The next chapter extends this foundation by introducing an intelligent decision layer, enabling the platform to anticipate user behavior and improve ride matching through predictive and algorithmic approaches.

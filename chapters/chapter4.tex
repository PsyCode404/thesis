\chapter{Ride Management and Real-Time Interaction}

\section{Introduction}

Following the implementation of authentication and user management, this chapter introduces the operational core of the system: ride management and real-time interaction.

This layer is responsible for orchestrating all ride-related operations while ensuring consistency, constraint enforcement, and synchronization across all system components.

Two complementary concerns are addressed:

\begin{itemize}
    \item \textbf{State Management}: controlling ride creation, booking, and lifecycle transitions under strict business rules.
    \item \textbf{State Propagation}: ensuring that all participants observe a consistent and up-to-date system state in real time.
\end{itemize}

To address these concerns, the implementation is structured into two complementary development phases:

\begin{itemize}
    \item \textbf{Sprint 3 — Ride Management}: focuses on modeling entities, enforcing constraints, and managing ride lifecycle.
    \item \textbf{Sprint 4 — Real-Time Interaction}: introduces an event-driven communication layer for live synchronization.
\end{itemize}

This separation ensures a clear distinction between decision logic and data propagation, improving modularity and maintainability.

\subsection{Sequence Diagram — Ride Booking}

\begin{figure}[H]
\centering
\includegraphics[width=0.9\textwidth]{images/diagrams/sequence_booking.png}
\caption{Sequence Diagram — Ride Booking Process}
\end{figure}

The booking process is implemented as a strictly controlled validation pipeline executed entirely at the backend level.

The client initiates the request, but all decision-making logic is centralized in the backend. This ensures that constraints cannot be bypassed by client-side manipulation.

The validation sequence follows a deterministic order:

\begin{itemize}
    \item Retrieval of the ride from the database to ensure existence.
    \item Verification of seat availability to prevent overbooking.
    \item Enforcement of role constraints to prevent users from booking their own rides.
    \item Validation of temporal constraints to ensure the ride is still valid.
\end{itemize}

Each validation step acts as a gate. If any condition fails, the process is immediately terminated and an error response is returned.

Only when all constraints are satisfied does the system create the reservation and update seat availability. This guarantees that the system never enters an inconsistent state.

This sequence demonstrates a key architectural principle: \textbf{all business logic is enforced server-side}, while the client remains a thin interaction layer.

\subsection{Ride Lifecycle Management}

\begin{figure}[H]
\centering
\includegraphics[width=0.85\textwidth]{images/architecture/ride_lifecycle.png}
\caption{Ride Lifecycle State Machine}
\end{figure}

The ride lifecycle is modeled as a finite state machine to ensure controlled and predictable state transitions.

Each state corresponds to a distinct phase of ride execution, and transitions are triggered either by user actions or system rules.

\begin{itemize}
    \item Scheduled → Driver En Route: initiated by the driver.
    \item Driver En Route → Arrived: triggered when the driver reaches the pickup location.
    \item Arrived → In Progress: triggered when the ride starts.
    \item In Progress → Completed: triggered when the ride ends.
\end{itemize}

Two additional states handle edge conditions:

\begin{itemize}
    \item Cancelled: can be triggered at any stage by user action.
    \item Missed: automatically triggered when the ride is not started within the allowed timeframe.
\end{itemize}

This model ensures that invalid transitions are impossible and that the system always reflects a coherent state.

\section{Sprint 4: Real-Time Interaction}

\subsection{Real-Time Architecture}

To ensure immediate synchronization between system components, the platform adopts a WebSocket-based communication architecture.

Unlike traditional HTTP communication, which follows a request-response model, WebSocket establishes a persistent bidirectional connection between the client and the server. Once the connection is initiated, it remains open, allowing the backend to push updates asynchronously without requiring repeated client requests.

This architectural choice is motivated by the need for:

\begin{itemize}
    \item Low-latency communication
    \item Continuous synchronization of ride state
    \item Efficient propagation of frequent updates (e.g., location tracking)
\end{itemize}

\subsubsection{Connection Lifecycle}

The real-time communication process begins when the mobile application initiates a WebSocket connection with the backend server.

This connection is associated with the authenticated user session, ensuring that events are routed only to relevant participants.

Once established, the connection remains active throughout the ride lifecycle, enabling continuous communication without additional overhead.

\subsubsection{Event-Driven Communication Model}

The system operates on an event-driven model.

Whenever a significant change occurs within the system, such as:

\begin{itemize}
    \item ride status updates,
    \item driver location updates,
    \item reservation confirmations,
    \item notification triggers,
\end{itemize}

the backend generates an event that is immediately transmitted through the WebSocket server.

Clients subscribed to the corresponding ride or user channel receive these events in real time and update their interface accordingly.

This approach eliminates the need for polling and ensures that all users observe the same system state with minimal delay.

\subsubsection{Multi-Client Synchronization}

A key advantage of this architecture is its ability to synchronize multiple clients simultaneously.

For a given ride, several participants may be connected at the same time (driver and multiple passengers). When an event occurs, it is broadcast to all relevant clients, ensuring that:

\begin{itemize}
    \item ride status remains consistent across all devices,
    \item location updates are reflected simultaneously,
    \item user interfaces remain aligned with backend state.
\end{itemize}

This guarantees a coherent shared experience across the platform.

\subsection{Integration with Ride Lifecycle}

The real-time communication layer is tightly coupled with the ride lifecycle defined in the previous sprint.

Each transition within the lifecycle triggers a corresponding event, which is propagated through the WebSocket infrastructure. For example:

\begin{itemize}
    \item When the driver starts the ride, a status update is broadcast.
    \item When the driver reaches the pickup point, an arrival event is emitted.
    \item When the ride is completed, all participants are notified instantly.
\end{itemize}

This integration ensures that lifecycle transitions are not only enforced at the backend level but are also immediately reflected in the user interface.

As a result, the system maintains a consistent and synchronized representation of ride execution across all participants.

\subsection{Notification System}

The notification system complements the real-time architecture by transforming backend events into structured, user-facing interactions.

While WebSocket ensures the delivery of real-time updates, the notification layer provides a persistent and interactive representation of these events.

\subsubsection{Role of Notifications}

Notifications are designed to:

\begin{itemize}
    \item Inform users of important ride-related events in real time
    \item Provide immediate feedback on system actions
    \item Guide users toward relevant interactions within the application
    \item Maintain awareness even when users are not actively monitoring a ride
\end{itemize}

\subsubsection{System Flow}

The notification mechanism follows a structured pipeline:

\begin{enumerate}
    \item A system event occurs (e.g., reservation accepted, ride status updated)
    \item The backend generates and stores a notification entry in the database
    \item A WebSocket event is emitted to the concerned user(s)
    \item The mobile application receives the event in real time
    \item The notification is displayed and the interface is updated
\end{enumerate}

This dual mechanism ensures both persistence (through database storage) and immediacy (through real-time delivery).

\subsubsection{Types of Notifications}

The platform supports multiple categories of notifications:

\begin{itemize}
    \item Ride lifecycle updates (ride started, driver arrived, ride completed)
    \item Reservation updates (accepted or rejected requests)
    \item Boarding alerts (upcoming or missed boarding)
    \item System alerts (ride cancellation, availability changes)
\end{itemize}

Each notification is associated with contextual data, allowing precise interaction and navigation.

\subsubsection{User Interaction and Navigation}

Notifications are designed as interactive elements rather than passive messages.

Users can:

\begin{itemize}
    \item Access notifications through a dedicated interface displaying a chronological list
    \item Mark notifications as read individually or in bulk
    \item Navigate directly to relevant application screens by interacting with notifications
\end{itemize}

The navigation behavior depends on the notification context:

\begin{itemize}
    \item Ride-related notifications redirect to the \textit{Ride Details Screen}
    \item Reservation-related notifications redirect to the \textit{Reservation Details Screen}
    \item Boarding or pickup notifications redirect to the active ride interface
\end{itemize}

This design integrates notifications into the application's navigation flow, enabling seamless transitions between events and actions.

\subsection{System Observability and Administrative Interface}

In addition to user-facing functionalities, the system incorporates an administrative observability layer that provides real-time visibility into system behavior and operational state.

This layer is not designed as a separate control module, but rather as a direct extension of the event-driven architecture described previously. It leverages the same real-time infrastructure to expose system activity, enabling administrators to monitor, analyze, and understand system dynamics as they occur.

\subsubsection{System Event Logging}

At the core of this layer lies a structured event logging mechanism.

Every significant action within the system—such as user authentication, ride creation, reservation updates, and lifecycle transitions—is recorded as a \textit{system event}. These events are generated at the backend level and persisted in a dedicated data model, ensuring traceability and historical consistency.

This approach guarantees that all system interactions are captured in a standardized format, enabling precise tracking of user activity and system behavior over time.

\section{Conclusion}

This chapter presented the implementation of a complete ride management system enhanced by a real-time communication architecture.

The integration of backend-controlled business logic, a structured ride lifecycle, and a WebSocket-based event-driven system ensures both consistency and responsiveness across the platform. Each system component, from ride state transitions to user-facing notifications, is tightly synchronized, enabling all participants to maintain a shared and up-to-date view of ongoing rides.

The introduction of the notification system further reinforces this architecture by transforming system events into actionable user interactions, effectively bridging backend processes and frontend behavior.

Overall, this chapter establishes a robust and reactive system capable of handling dynamic ride execution scenarios while maintaining data integrity and user awareness.

However, while the system efficiently responds to user actions and real-time events, it remains fundamentally reactive. The next chapter builds upon this foundation by introducing an intelligent decision layer, enabling the system to anticipate user needs and optimize ride matching through algorithmic and predictive approaches.
